# v0.2 Maintainer Workspace Plan

## Goal

Ship a stronger `v0.2` foundation for OSS Maintainer Copilot by turning the current workflows into a more reusable maintainer workspace.

This milestone should make the repository easier to trust, easier to extend, and easier for contributors to enter safely.

## Release Outcomes

By the end of `v0.2`, the project should have:

- a deeper shared repository intelligence layer
- onboarding maps with more tailored contributor guidance
- issue triage internals split into clearer extension points
- stronger PR brief signals for changed areas and noisy pull requests
- harder-to-break release note collection
- broader evaluation fixtures and regression tests
- workflow contract tests for GitHub Actions
- documented skill boundaries for recurring maintainer workflows

## Guiding Constraints

- Keep the package repo-agnostic.
- Prefer deterministic logic over opaque behavior.
- Make schema evolution additive when possible.
- Keep public examples and workflow outputs aligned with code changes.
- Favor maintainability and contributor readability over clever abstractions.

## Workstreams

### Workstream 1: Shared Repository Intelligence

Strengthen `repo_intel` so it becomes the base layer for onboarding, repo summaries, and future maintainer automation.

#### Issue 1: Deepen Repository Profile Extraction

- Suggested title: `Deepen repository intelligence extraction for setup, docs, and workflow areas`
- Goal: infer richer setup paths, major areas, and workflow hints from normalized repository inputs.
- Primary files:
  - `src/oss_maintainer_copilot/agents/repo_intel.py`
  - `src/oss_maintainer_copilot/schemas/repo_intel.py`
  - `tests/fixtures/repos/`
  - `tests/`
- Deliverables:
  - better setup-step extraction from README and CONTRIBUTING content
  - improved area grouping for `src/`, `tests/`, `examples/`, `docs/`, and `.github/workflows/`
  - more reliable maintainer workflow inference
  - stronger confidence scoring based on available repo context
- Acceptance criteria:
  - richer repository intelligence output on at least 3 fixture styles
  - no breaking schema changes without explicit migration docs
  - fixture-backed tests for sparse, medium, and rich repo profiles

#### Issue 2: Add Maintainer-Facing Repository Summaries

- Suggested title: `Add maintainer-facing repository summaries on top of repository intelligence`
- Goal: provide a fast repo brief for maintainers who need context before triage, review, or onboarding work.
- Primary files:
  - `src/oss_maintainer_copilot/agents/repo_intel.py`

- Acceptance criteria:
  - repo intelligence output is useful for both contributors and maintainers
  - checked-in examples remain easy to scan

## Release Gate For v0.2

Do not call `v0.2.0` done until:

- repository intelligence and onboarding have multi-fixture coverage
- triage internals are split into clearer layers
- PR briefs handle changed areas and noisy PRs better
- release drafting has broader fixture coverage
- examples match the current tested behavior
- GitHub workflow behavior has explicit contracts

## Notes For GitHub Issue Creation

When you turn these into GitHub issues:

- keep each issue scoped to one workstream outcome
- link fixture updates directly to code changes
- prefer additive schema changes over renames
- call out public-output changes explicitly in the issue body
- note whether the issue is safe for a new contributor or needs maintainer review
